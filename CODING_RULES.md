# Правила і команди для A11y Oracle проекту

## 📋 Загальні принципи

### 1. ✅ ЗАСТОСОВУЄМО (корисні для нас)

#### 🔍 **startcheck** - Перевірка перед дією
**Що:** Перевіряти контекст перед виконанням команд
**Коли:** Перед зміною файлів, запуском скриптів, видаленням

**Приклад:**
```bash
# ❌ Погано - одразу видаляємо
rm worker.py

# ✅ Добре - спочатку перевіряємо
ls -la worker.py  # Перевірити що файл існує
cat worker.py | head -20  # Подивитись що всередині
rm worker.py  # Тепер видаляємо
```

**Для Python:**
```bash
# Перед запуском - перевірити синтаксис
python3 -m py_compile worker.py
# Якщо OK - запускати
python3 worker.py
```

---

#### 🐛 **kaki** - Аналіз помилок
**Що:** Коли щось не працює - аналізувати причину і як запобігти в майбутньому
**Коли:** Після кожної помилки

**Приклад:**
```
Помилка: Worker зациклився

Аналіз (kaki):
1. Причина: Не перевіряв чи job вже обробляється
2. Як виправити: Додати перевірку current_job
3. Як запобігти: Завжди очищати стан після обробки
4. Тест: Створити 3 різні job і перевірити

Результат: Додано перевірку дублікатів + timeout
```

---

#### 📝 **REPLACE_BAD_CODE_BY_FULL_FILE** - Заміна цілого файлу
**Що:** Якщо багато змін або ризик зламати - замінити файл повністю
**Коли:** >3 змін в одному файлі, складна логіка, ризик конфлікту

**Приклад:**
```python
# ❌ Погано - 5 str_replace підряд
fs_write str_replace line 10
fs_write str_replace line 25
fs_write str_replace line 40
fs_write str_replace line 55
fs_write str_replace line 70

# ✅ Добре - один раз весь файл
fs_write create worker.py [повний код з усіма змінами]
```

**Коли НЕ використовувати:**
- Файл >500 рядків
- Зміна 1-2 рядків
- Простий bugfix

---

#### 🤖 **selfself** - Автономність
**Що:** Шукати рішення самостійно, не питати користувача без потреби
**Коли:** Завжди

**Приклад:**
```
# ❌ Погано
"Який Python використовувати?"
"Де створити файл?"
"Яку назву дати змінній?"

# ✅ Добре
which python3  # Знайти сам
pwd  # Визначити поточну директорію
# Використати логічну назву (worker.py, api_server.py)
```

**Коли питати:**
- Вибір архітектури (сервер vs локально)
- Бізнес-логіка (скільки платити за перевірку?)
- Дані користувача (email, API ключі)

---

#### 📦 **scriptsmany** - Автоматизація через скрипти
**Що:** Якщо треба виконати >3 команди - створити .sh скрипт
**Коли:** Повторювані дії, складні команди, багато кроків

**Приклад:**
```bash
# ❌ Погано - виконувати вручну
execute_bash "curl ..."
execute_bash "python3 ..."
execute_bash "gcloud ..."
execute_bash "systemctl ..."

# ✅ Добре - один скрипт
fs_write create deploy.sh [всі команди]
execute_bash "./deploy.sh"
```

**Коли створювати скрипт:**
- >3 команди підряд
- Буде використовуватись багато разів
- Складна логіка (if/else, loops)

**Коли НЕ створювати:**
- 1-2 прості команди
- Одноразова дія
- Швидка перевірка

---

#### 🎯 **absolutefull** - Абсолютні шляхи
**Що:** Завжди використовувати повні шляхи до файлів
**Коли:** Завжди в скриптах, конфігах, systemd services

**Приклад:**
```bash
# ❌ Погано - відносні шляхи
python3 worker.py
source venv/bin/activate
./script.sh

# ✅ Добре - абсолютні шляхи
/usr/bin/python3 /home/butenhome/aiwork/worker.py
source /home/butenhome/aiwork/venv/bin/activate
/home/butenhome/aiwork/script.sh
```

**Особливо важливо:**
- systemd service файли
- cron jobs
- скрипти які запускаються з різних директорій

---

#### 📚 **applystyle** - Прикладне навчання
**Що:** Пояснювати складні речі через: Мета → Терміни → Приклади → Градієнт
**Коли:** Пояснення нових концепцій, технологій, архітектури

**Структура:**
1. **Мета (Зачем?)** - Навіщо це потрібно?
2. **Глосарій** - Визначення термінів + 3 приклади
3. **Градієнт** - Від простого до складного
4. **Масса** - Конкретні приклади, код
5. **Баланс** - Переваги vs недоліки

**Приклад:**
```markdown
# Що таке Worker?

## Мета
Автоматизувати рутину (відкривання браузера, відправка результатів)

## Глосарій
- Worker = програма-помічник
- Async = робити кілька речей одночасно
- Hotkey = гаряча клавіша

## Градієнт
1. Простий приклад: відкрити один сайт
2. Середній: обробити чергу
3. Складний: async + timeout + error handling

## Масса (код)
[конкретний код з коментарями]

## Баланс
Переваги: швидко, зручно
Недоліки: треба налаштувати
```

### 2. ⚠️ ЧАСТКОВО ЗАСТОСОВУЄМО (адаптуємо)

#### 📊 **logexplane** - Логування
**Оригінал:** Логувати в /mnt/c/vk/session_log.txt
**Наша адаптація:** Логувати в ~/aiwork/logs/

**Приклад:**
```python
import logging

logging.basicConfig(
    filename='/home/butenhome/aiwork/logs/worker.log',
    level=logging.INFO,
    format='%(asctime)s - %(message)s'
)

logging.info(f"Обробка job {job_id}")
logging.error(f"Помилка: {e}")
```

**Коли використовувати:**
- Дебаг складних проблем
- Моніторинг production
- Аналіз продуктивності

## 💡 Золоті правила

1. **Перевіряй перед дією** (startcheck)
2. **Аналізуй помилки** (kaki)
3. **Автоматизуй рутину** (scriptsmany)
4. **Будь автономним** (selfself)
5. **Використовуй повні шляхи** (absolutefull)
6. **Пояснюй структуровано** (applystyle)
7. **НЕ запускай інтерактивні програми** (noblock)

---

#### 🚫 **noblock** - Не блокувати виконання
**Що:** Ніколи не запускати через execute_bash програми які чекають input
**Коли:** Worker, сервери, GUI програми, інтерактивні скрипти

**Приклад:**
```bash
# ❌ НІКОЛИ - я завишу
execute_bash "python3 worker.py"
execute_bash "nano file.txt"
execute_bash "top"

# ✅ ЗАВЖДИ - тільки інструкції
"Запусти в окремому терміналі: python3 worker.py"
"Відкрий файл: nano file.txt"
```

**Виключення (можна запускати):**
- Швидкі команди (<5 сек): ls, cat, curl
- Команди з timeout: curl --max-time 5
- Фонові процеси: nohup python3 worker.py &
